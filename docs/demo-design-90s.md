## Core Narrative Assets (the narrative assets broken out)

- Brand line: "**Bonsai — fed by your life, tending it back.**" (revised 2026-07-05: the original "apps you tend, not apps you build" had the direction backwards — it's not the user tending the app, it's the app tending the user)

### Philosophy layer (why this product holds up)

1. **PARA: context is organized by "purpose", not by "tool".** In traditional software your job-hunt context is scattered across email, calendar, spreadsheets, and chat — the tools hold the data, but no tool holds the "goal". Bonsai assigns context to goals (Projects/Areas/Resources/Archive); the goal is the first-class citizen.
2. **The human's role migrates: from feeding context to only making decisions.** In the ideal state the AI advances your goals autonomously 24 hours a day. Early on the human provides context (Day 1's AI-question confirmation area is exactly this); after it has run for a while the human only makes decisions (Day 90's suggestion banner + decision cards are exactly this). **The UI's evolution is not decoration — it is the visualization of this role migration.**
3. **A direct response to the RFS text.** Ankit Gupta: "my email client looks more like a task list, and a student's looks more like an events calendar... software companies will ship these shared primitives with full intention that users will heavily modify the final interfaces." — The frozen component pool is the shipped primitives, and generating interfaces per goal is the heavily modified final interfaces. The RFS talks about different people getting different interfaces; Bonsai goes a step further: every goal of the same person gets its own interface. This passage can be quoted verbatim in the pitch.

## Four differentiation directions (all written out in full; the video only fits one main thread — the rest must land in the written submission materials)

### Direction 1: A living app vs one-shot generation (vs Lovable / v0 / Replit)

**Claim**: Lovable/v0 are "generate and deliver" — they produce code, and then the software goes static and starts rotting like traditional software. Bonsai's generation is not a one-time event but a continuous process: the context changes and the app changes with it. Day 90 is not the same interface as Day 1, yet the user did nothing.

**Evidence**: demo state 1 → state 2, `warm.py --evolve` incremental evolution, the same persona's cache file carrying over (`cache.jobhunt.json` unchanged, the pack swapped to day90).

**One-liner**: "App builders generate software once. Ours keeps growing with your life."

**Defensive phrasing (goes into the written submission; judges have no follow-up round)**: "Isn't that just regenerating it all over again?" — No, it is not full regeneration: evolution is incremental (only the first-view is regenerated), the layout skeleton stays continuous, and the user's spatial memory is not broken. What changes is the content and the widgets; what stays constant is the feeling that "this is my app."

### Direction 2: Structural safety — the model never generates code (capability lock)

**Claim**: every "AI-generated UI" approach faces the same problem: arbitrary code execution won't make it into the App Store and won't pass an enterprise security audit. Bonsai's model outputs only DSL structure, composing whitelisted components from the frozen component pool; rendering always happens on-device. **This is an architecture that can actually ship to consumers' phones**, not a toy inside a browser sandbox.

**Evidence**: the component whitelist in `bridge/system_prompt.txt`; the depth cap enforced on-device (deeper links smuggled into the DSL are simply ineffective); on DSL parse failure the previous screen is kept — never a blank screen.

**One-liner**: "The model composes structure, never code — that's how this ships to the App Store."

**This is also a direct answer to the RFS**: the RFS asks "How will a developer make software that can be accessed by the user's coding agents?" — Answer: ship primitives + a capability lock, not source code.

### Direction 3: goal-native, not tool-native (vs ChatGPT / Notion)

**Claim**: ChatGPT can give advice but doesn't hold your goal state — every conversation starts from zero. Notion/calendars hold data but don't advance goals — they are warehouses, not engines. Bonsai organizes context by purpose, the AI advances things autonomously, and the human shifts from "someone operating tools" to "someone making decisions."

**Evidence**: the PARA shell itself; the Day 1 confirmation area (the AI proactively asks for the context it lacks) → the Day 90 suggestion list (the AI proactively offers decision options).

**One-liner**: "Tools hold your data. Bonsai holds your goals."

**Defensive phrasing (goes into the written submission; judges have no follow-up round)**: "How is this different from AI agent products?" — Agent products run inside a chat window; a goal deserves an interface of its own. The interface is decision efficiency: blood glucose should be a curve, a job hunt should be a pipeline, and chat bubbles express neither well.

**Defensive phrasing ("Isn't this just Obsidian/Notion plus AI?"):** Obsidian's Dataview/Bases ecosystem has long been able to turn notes into "living dashboards", but that is a query language — only people who can write Dataview queries and wrangle Templater templates can actually use it. You yourself are that forward deployed engineer; the AI at best helps you generate that query code, and the configuration and maintenance responsibility stays with you the whole time. Bonsai flips it: the end user doesn't need to know any query syntax — one sentence gets you an interface — and the model outputs only structured DSL, never code, with rendering always on-device. That is the architecture that fits into an ordinary person's phone and passes App Store review, a form the Obsidian plugin ecosystem cannot replicate. The difference is not as thin as "no-code"; it is no-engineering: someone who can't write a query language and a power Obsidian user get exactly the same result out of Bonsai.

One layer deeper: when someone manages to build Obsidian into a living dashboard with AI, it was never just "knowing how to use AI" — it takes two skills at once: **FDE skills** (understanding Dataview queries, managing Claude Code, writing skills/cronjobs/hooks to drive it) and **FDPM skills** (knowing what structure, what views, what priorities to design for your own job hunt, health, and life stage). Ordinary people have neither — installing an "AI-enhanced Obsidian" won't magically grant them the FDE engineering ability, nor the FDPM product judgment. This is exactly what the RFS text means by "these primitives will be built and shipped by a software team": the design judgment should live in the frozen component pool and system prompt the developer ships, not be demanded of every end user, who would then need to be both an engineer and a product designer. Bonsai packages both scarce skills — FDE and FDPM — into the product itself.

A concrete example: the "AI-summarized periodic note" you built into your Obsidian home — daily/weekly notes get AI-summarized and placed into home — is essentially the hand-rolled version of the **"AI dreams nightly"** whoa moment written in the previous section of this doc. The difference: your version depends on you writing the cronjob/hook to decide when it runs, judging for yourself what information the summary should keep, and designing how home presents it; in Bonsai this pipeline (the health Area's sleep-data correlation, the morning insight card) is native product behavior that every user gets with zero configuration. This is the most direct evidence of two implementation paths for the same insight — more persuasive than abstractly talking about "FDE/FDPM skills" — and it is recommended to use it as the example directly in the written submission.

### Direction 4: A real native app, not a webpage/chat (technical moat)

**Claim** (user's own words): "Because we've already got it running, we simply are a real native app — we have an absolute technical advantage." In a 5-hour hackathon others can produce a web mockup or a chat wrapper; you have a Flutter app running on a real iPhone: tab navigation, native feel, offline caching (previously generated screens open even without a network), and three layers of caching making everything open instantly.

**Evidence**: the real-device screen recording itself is the evidence. Sweeping across tabs with every template opening instantly; deep navigation; smoothness in profile mode.

**One-liner**: "This isn't a mockup in a browser tab. It's a real app on a real phone, working today."

**The correct use of this direction**: it is not the narrative main thread, it is **texture** — record the whole video on a real device instead of a simulator/web recording, and the claim proves itself without a single word being said. The candidate narrative for the Biggest Engineering Lift award (the iOS async rabbit hole, the three-layer cache, the closed-app architecture) also hangs under this direction — with no follow-up round for the judges, this must go into the written submission's technical overview, or the judges will have no way to learn about it.

## Moat: keep an angle of 20+ degrees from the model vendors (user principle)

**Principle** (user's own words): "Keep at least a 20-degree angle from what Anthropic builds — perpendicular is best. Never collide head-on with the model vendors."

**Why Bonsai is naturally close to perpendicular**:

- **Model vendors build the horizontal layer**: Anthropic/OpenAI build general agent capabilities (Claude Code, general assistants, computer use) — their product is "intelligence" itself, sold to everyone. Bonsai builds the vertical layer: a **personal goal operating system**. The model is a supplier, not a competitor — the bridge uses `claude -p` today and can swap in any model tomorrow, with the narrative unchanged
- **The moat isn't inside the model, it's outside it**: three things the model vendors won't build and can't build:
  1. **The goal-context organization layer** (PARA packs): the user's life context accumulates, structured by purpose, growing thicker with use — this is a data asset, not a model capability
  2. **The frozen component pool / design system** (shipped primitives): the RFS says software companies will "ship shared primitives" — that is product-company work: design taste + domain components; model vendors have no incentive to build a component pool for some vertical
  3. **Accumulated personalized state**: Day 90 is more valuable than Day 1 because 90 days of context and decision history have settled into this goal. The switching cost rises over time
- **Model progress is a tailwind, not a threat**: the stronger the model, the higher the DSL generation quality and the lower the cost — Bonsai is a **consumer** of frontier models, and every model-vendor release is a free upgrade

**Defensive phrasing (goes into the written submission; judges have no follow-up round)**: "Anthropic ships artifacts / Claude can generate UI — what do you do?" — What they generate is one-off UI in a browser sandbox: no goal state, no evolution, no native distribution. Our relationship to them is Shopify's relationship to AWS: use their compute, build the vertical they don't

**Pitch one-liner**: "We never compete with model vendors — we're vertical. Models are our suppliers; our moat is the goal-context layer and the primitives they compose."

## Approaches Considered (candidate 90-second narrative structures)

### Approach A: "One life, one living app" (timeline narrative) ⭐ SELECTED

- **Arc**: hook (tagline + one-sentence problem) → **onboarding** (the user states their goals, the app grows the PARA skeleton on the spot) → job seeker Day 1 (confirmation area, human feeds context) → **Day 90 evolution** (the same app grew on its own, the human only makes decisions) → live intent generation (the user can still grow a new screen at any time) → close (3-second cross-goal comparison flash + one market line)
- **Lead differentiation**: Direction 1 (living app), Direction 3 as the voiceover logic, Direction 4 conveyed implicitly through real-device texture
- **Seconds budget** (target 83s, keeping 7s of headroom — 90s is a hard cap, and editing right up to the limit leaves zero tolerance): hook 6s → onboarding 12s → Day1 14s → Day90 22s → intent generation 16s → comparison+close 13s
- **Effort**: M (three footage segments + editing) · **Risk**: Low (all footage covered by the runbook)
- **Pros**: the narrative has a temporal arc, telling the "software is alive" story nobody else has; both selected whoa moments are the protagonists; the philosophy layer (role migration) grows naturally inside the narrative
- **Cons**: the "90 days" of Day 1 → Day 90 needs voiceover to explain it is a simulated evolution — handled badly it looks like cheating; the comparison shot gets squeezed to 3 seconds

### Approach B: "One app, a thousand faces for one person" (spatial-contrast narrative)

- **Arc**: hook → side-by-side split screen: the same person's job-hunt Project home screen vs health Area home screen (same binary, same component pool) → quote the RFS text ("email client vs events calendar — we do it even across one person's different goals") → evolution quick flash → close
- **Lead differentiation**: Direction 3 (goal-native) + the highest RFS hit rate
- **Effort**: M · **Risk**: Low
- **Pros**: the strongest visual contrast within the opening 10 seconds; the most literal response to the RFS, the least likely to be missed by the judges
- **Cons**: the contrast is static ("different goals, different interfaces" is something competitors could also claim), giving up the unique "alive/growing" narrative; the user's chosen whoa moments become supporting cast

**DECISION (2026-07-05, user's call): Approach A.** B's strongest shot (the side-by-side comparison) becomes a 3-second insert at the end of A. The former Approach C (technical-credibility narrative) has been deleted — a hackathon video doesn't need defensive technical justification; real-device texture is A's default way of shooting and speaks for itself, and the engineering narrative (capability lock, the iOS async rabbit hole, the three-layer cache) goes only into the written submission.
