### 哲学层(为什么这个产品成立)

1. **PARA:context 按"目的"组织,不是按"工具"组织。** 传统软件里你的求职 context 散在邮箱、日历、表格、聊天里——工具持有数据,但没有工具持有"目标"。Bonsai 把 context 归属到 goal(Projects/Areas/Resources/Archive),goal 才是第一公民。
2. **人的角色迁移:从喂 context 到只做决策。** 理想状态下 AI 24 小时自主推进目标。早期人提供 context(Day 1 的 AI 提问确认区就是这个),运行一段时间后人只做决策(Day 90 的建议 Banner + 决策卡就是这个)。**UI 的演化不是装饰,它是这个角色迁移的可视化。**
3. **直接回应 RFS 原文。** Ankit Gupta:"my email client looks more like a task list, and a student's looks more like an events calendar... software companies will ship these shared primitives with full intention that users will heavily modify the final interfaces." —— 冻结组件池就是 shipped primitives,按 goal 生成界面就是 heavily modified final interfaces——RFS 讲的是不同人得到不同界面,Bonsai 更进一步:同一个人的每个 goal 都得到自己的界面。这段可以在 pitch 里逐字引用。

### 健康 Area 深化(2026-07-05 加):iOS Health 互通 + 夜间 "dream"

- **context 不只来自人喂**:健康 Area 直接与 iOS HealthKit 互通,睡眠、步数等数据自动流入 goal context——把方向 3"工具持有数据,Bonsai 持有目标"从主张变成事实

## 四个差异化方向(全部写透;视频只容一条主线,其余必须落进书面提交材料)

### 方向 1:活的 app vs 一次性生成(vs Lovable / v0 / Replit)

**主张**:Lovable/v0 是"生成即交付"——产出代码,然后软件就静止了,和传统软件一样开始腐烂。Bonsai 的生成不是一次事件,是持续过程:context 变,app 跟着变。Day 90 和 Day 1 不是同一个界面,但用户什么都没做。

**证据**:demo 状态 1 → 状态 2,`warm.py --evolve` 增量演化,同一 persona 的缓存文件延续(`cache.jobhunt.json` 不变,pack 换成 day90)。

**一句话版本**:"App builders generate software once. Ours keeps growing with your life."

**防御表述(写进书面提交,评审无追问环节)**:"那不就是重新生成一遍吗?" —— 不是全量重生成:演化是增量的(只重生成 first-view),布局骨架保持连续性,用户的空间记忆不被破坏。变的是内容和控件,不变的是"这是我的 app"的感觉。

### 方向 2:结构化安全 —— 模型永不生成代码(capability lock)

**主张**:所有"AI 生成 UI"的方案都面临同一个问题:任意代码执行进不了 App Store,也过不了企业安全审计。Bonsai 的模型只输出 DSL 结构,组合冻结组件池里的白名单组件;渲染永远在设备端。**这是能真正 ship 到消费者手机上的架构**,不是浏览器沙箱里的玩具。

**证据**:`bridge/system_prompt.txt` 的组件白名单;设备端深度上限强制执行(DSL 里混进更深的链接也无效);DSL 解析失败保留上一屏,永不白屏。

**一句话版本**:"The model composes structure, never code — that's how this ships to the App Store."

**这也是对 RFS 的直接回答**:RFS 问 "How will a developer make software that can be accessed by the user's coding agents?" —— 答案:交付 primitives + capability lock,不是交付源码。

### 方向 3:goal-native,不是 tool-native(vs ChatGPT / Notion)

**主张**:ChatGPT 能给建议,但不持有你的 goal state——每次对话从零开始。Notion/日历持有数据,但不推进目标——它们是仓库,不是引擎。Bonsai 按目的组织 context,AI 自主推进,人从"操作工具的人"变成"做决策的人"。

**证据**:PARA shell 本身;Day 1 确认区(AI 主动问它缺的 context)→ Day 90 建议列表(AI 主动给决策选项)。

**一句话版本**:"Tools hold your data. Bonsai holds your goals."

**防御表述(写进书面提交,评审无追问环节)**:"和 AI agent 产品的区别?" —— agent 产品在聊天窗口里跑;goal 值得拥有一个属于它的界面。界面就是决策效率:血糖该是曲线,求职该是 pipeline,聊天气泡两者都表达不好。

### 方向 4:真 native app,不是网页/chat(技术护城河)

**主张**(用户原话):"因为我们已经跑通了,所以我们就是真 native app,我们在技术上有绝对优势。" 5 小时 hackathon 里别人能做出网页 mockup 或 chat wrapper;你们有一个在真 iPhone 上跑的 Flutter app:tab 导航、原生手感、离线缓存(生成过的屏幕断网也能开)、三层缓存让一切秒开。

**证据**:真机录屏本身就是证据。扫 tab 全模板秒开;深度导航;profile 模式流畅度。

**一句话版本**:"This isn't a mockup in a browser tab. It's a real app on a real phone, working today."

**这条的正确用法**:不是叙事主线,是**质感**——整个视频用真机录屏而不是模拟器/网页录屏,这条主张就不言自明,一个字都不用说。Biggest Engineering Lift 奖的候选叙事(iOS async 深坑、三层缓存、封闭 app 架构)也挂在这条下面——评审无追问,这段必须写进书面提交的技术概要,否则评委无从得知。

## 护城河:与模型厂商保持 20 度以上的夹角(用户原则)

**原则**(用户原话):"至少要和 Anthropic 做的东西有 20 度以上的夹角,最好就是垂直了才好,永远不要和这些模型厂商正面碰撞。"

**为什么 Bonsai 天然接近垂直**:

- **模型厂商做的是水平层**:Anthropic/OpenAI 做通用 agent 能力(Claude Code、通用助手、computer use)——它们的产品是"智能"本身,卖给所有人。Bonsai 做的是垂直层:**个人目标操作系统**。模型是供应商,不是竞争对手——bridge 今天用 `claude -p`,明天可以换任何模型,叙事不变
- **护城河不在模型里,在模型外**:三样东西模型厂商不会做也做不了:
  1. **goal-context 的组织层**(PARA packs):用户的人生 context 按目的结构化积累,越用越厚——这是数据资产,不是模型能力
  2. **冻结组件池 / 设计系统**(shipped primitives):RFS 说软件公司会"ship shared primitives"——这是产品公司的活,设计品味 + 领域组件,模型厂商没有动机做某个垂直领域的组件池
  3. **积累的个人化状态**:Day 90 比 Day 1 有价值,是因为 90 天的 context 和决策历史沉淀在这个 goal 里。切换成本随时间上升
- **模型进步是顺风不是威胁**:模型越强,DSL 生成质量越高、成本越低——Bonsai 是前沿模型的**消费者**,模型厂商的每次发布都是免费升级

**防御表述(写进书面提交,评审无追问环节)**:"Anthropic 发布了 artifacts / Claude 能生成 UI,你们怎么办?" —— 它们生成的是浏览器沙箱里的一次性 UI,没有 goal state、没有演化、没有 native 分发。我们和它们的关系是 Shopify 和 AWS 的关系:用它的算力,做它不做的垂直

**pitch 一句话版本**:"We never compete with model vendors — we're vertical. Models are our suppliers; our moat is the goal-context layer and the primitives they compose."

## Approaches Considered(90 秒叙事结构备选)

### Approach A:"一个人生,一个活的 app"(时间轴叙事)⭐ 已选定

- **弧线**:hook(tagline + 问题一句话)→ **onboarding**(用户说出自己的 goals,app 现场长出 PARA 骨架)→ 求职者 Day 1(确认区,人喂 context)→ **Day 90 演化**(同一 app 自己长了,人只做决策)→ 现场 intent 生成(用户还能随时长出新屏)→ 收尾(3 秒跨 goal 对比闪现 + market 一句话)

- **秒数预算**(目标 83s,留 7s 余量——90s 是硬上限,顶格剪辑零容错):hook 6s → onboarding 12s → Day1 14s → Day90 22s → intent 生成 16s → 对比+收尾 13s

- **Pros**:叙事有时间弧度,讲的是"软件活着"这个别人没有的故事;两个已选 whoa 时刻都是主角;哲学层(角色迁移)自然长在叙事里
- **Cons**:Day 1 → Day 90 的"90 天"需要旁白解释是模拟演化,处理不好像作弊;对比镜头被压缩到 3 秒

### Approach B:"同一个 app,一人千面"(空间对比叙事)

- **弧线**:hook → 并排分屏:同一个人的求职 Project 首屏 vs 健康 Area 首屏(同 binary 同组件池)→ 引 RFS 原文("email client vs events calendar——我们连一个人的不同 goal 都做到了")→ 演化快闪 → 收尾
- **主打差异化**:方向 3(goal-native)+ RFS 命中度最高
- **Effort**: M · **Risk**: Low
- **Pros**:开场 10 秒就有最强视觉对比;对 RFS 的回应最字面、最不会被评委错过
- **Cons**:对比是静态的("不同 goal 不同界面"竞品也能声称),放弃了"活着/生长"这个独有叙事;用户选的 whoa 时刻变成配角

**DECISION(2026-07-05,用户拍板):Approach A。** B 的最强镜头(并排对比)作为 A 结尾的 3 秒插入。原 Approach C(技术可信度叙事)已删——hackathon 视频不需要防御性的技术辩解;真机质感是 A 的默认拍摄方式,不言自明,工程叙事(capability lock、iOS async 深坑、三层缓存)只进书面提交。
